## gauntlet v0.2.0

The first feature release since v0.1.0. Gauntlet grows from a serial merge
queue into a parallel, container-aware one: checks run as a dependency graph,
can build and run their own images, and each can get a private workspace —
all while staying a single-threaded, tick-stepped reconcile loop.

Everything here is backward compatible: an existing config keeps its
serial-single-workspace behavior until you opt in.

### Parallelism

- **Dependency-graph checks.** A check spec can now declare `after` edges and
  a per-candidate `max-parallel`, so independent checks overlap and dependents
  wait for their prerequisites. The graph is validated up front (unknown
  names, self-edges, duplicates, cycles) even at `max-parallel 1`.
- **Daemon-wide execution cap.** The operator's `max-executions` bounds
  concurrent check executions across every target and hook — the real host
  ceiling — while `max-parallel` widens a single candidate's slice of it.

### Executor profiles & candidate-built images

- **Named executor profiles.** The daemon defines execution profiles (host,
  container, mounts, caches, resource ceilings); a check just names one. Host
  capabilities stay operator-owned — the repo side can only select.
- **Candidate-built check images.** An `image "name" { command … }` block
  builds an image inside the merge trial and runs consumer checks in it, keyed
  on the build's **captured immutable identity** (a local image ID or a
  digest-pinned reference — never a mutable tag). A trial that changes its
  Dockerfile proves that change in the same merge transaction.

### Isolated workspaces

- **`workspace "isolated"`.** Opt in and every graph node — each check and each
  image build — gets its own private materialization of the exact trial tree,
  so parallel or `after`-related nodes never collide through the filesystem.
  `after` becomes verdict ordering, not shared dataflow. Absent, the default
  single shared writable export is unchanged.

### Git integration & correctness

- **Deterministic history mtimes.** `export { mtimes "history" }` stamps each
  exported path with its owning commit's timestamp, so mtime-sensitive build
  tools see stable, content-derived times across runs.
- **gc-safe trials.** In-flight trial chains are pinned against `git gc`, so a
  concurrent repack can't collect a commit a running check still needs.
- **Trial-merge refs + verification statuses.** Gauntlet publishes the trial
  merge ref and reports commit status against the exact tested SHA.
- **`GAUNTLET_GIT_DIR`.** Checks are handed the bare repo, so in-tree `git`
  queries resolve against the real object store.

### GitHub App authentication

- **Refreshable installation tokens.** Authenticate as a GitHub App:
  short-lived installation tokens are minted and refreshed automatically, with
  host-scoped credential injection for git and per-request tokens for status
  updates.

### Operations

- **Graceful drain.** `SIGTERM` now drains in-flight work before exit instead
  of dropping it; a `gauntlet drain` CLI and API let you drain (optionally with
  a deadline) and wait for completion.
- **Dashboard refresh.** Mobile-friendly tables, white cards, command echo,
  and per-run check ratios.
- **`gauntlet land`** infers its target and topic from context.

### History schema

The embedded history store migrates automatically to schema v11, adding
columns for blocked-by/wait timing, built-image identity, and per-node
workspace materialization cost. Older databases are migrated in place; new
columns back-fill to zero.

### Fixes

Drain deadline handling and `--wait`; trial-ref leaks and phantom "landed"
events; treesame-merge ownership, `export-ignore`, and NUL-framing in the
mtimes walk; a Slack lost-wakeup race; and hardened askpass host scoping.

below. Container image:

